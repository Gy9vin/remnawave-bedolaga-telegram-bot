# Точечные рассылки по клиенту (приложению) — дизайн

- **Дата:** 2026-06-26
- **Статус:** согласован, готов к плану
- **Репозитории:** бот (`remnawave-bedolaga-telegram-bot`) — основное; кабинет (`bedolaga-cabinet`) — UI рассылки

## Проблема / цель
Нужно делать **точечные рассылки по клиентскому приложению** пользователя (Happ, v2rayNG, Streisand, «incy» и т.д.) — «кто на чём сидит» и слать только им. Сейчас рассылка фильтрует только по подписке/тарифу; данных о клиенте в нашей БД нет — они живут в RemnaWave (HWID-устройства, поле `userAgent`).

## Решения (из обсуждения)
- Данные о клиенте **синхронизируем в нашу БД** (live-запрос к панели на каждую рассылку не масштабируется).
- Синхронизация: **фоновая авто раз в сутки + ручная кнопка** «Обновить список клиентов».
- Имя клиента — **автоматически из UA** (префикс до `/`/`(`), без курируемого словаря; сырой UA тоже храним.
- Карточку пользователя с клиентом в этот объём НЕ включаем (отдельно).

## Текущее состояние (найдено разведкой)
- Панель отдаёт устройства: `get_all_hwid_devices()` (`app/external/remnawave_api.py:1213`, балк, постранично ≤1000/стр) и `get_user_devices(uuid)` (`:1237`). Запись устройства содержит `userAgent`, `platform`, `deviceModel`, `hwid`, `userUuid`.
- Маппинг панель→наш юзер: `User.remnawave_uuid` (`models.py:1911`), `Subscription.remnawave_uuid` (`models.py:2155`).
- Рассылка: `app/services/broadcast_service.py`, роуты `app/cabinet/routes/admin_broadcasts.py`. Фильтры строятся в `GET /cabinet/admin/broadcasts/filters` (`filters`/`tariff_filters`/`custom_filters`, ~строки 258-310); цель рассылки — `target_type`/`target`. Получатели резолвятся по цели в user-запрос (напр. `User.status=='active'`).
- UI рассылки: `bedolaga-cabinet/src/pages/AdminBroadcastCreate.tsx`, клиент `src/api/adminBroadcasts.ts`.

## Дизайн

### 1. Данные (миграция)
Таблица `user_clients`:
- `id` PK
- `user_id` FK→`users.id` (index)
- `app_name` String(64) — распознанное имя клиента (напр. `Happ`)
- `last_seen_at` DateTime|None — последняя активность устройства этого клиента (из `updatedAt`/`createdAt` устройства)
- `updated_at` DateTime — когда строка обновлена синком
- UNIQUE `(user_id, app_name)`, index на `app_name`.

Один пользователь → несколько строк (несколько клиентов).

### 2. Парсер UA
`parse_client_app(user_agent: str | None) -> str` (новый util, напр. `app/utils/client_detect.py`):
- берёт часть строки до первого `/` или `(` или пробела, тримит;
- нормализует регистр для группировки (напр. сохранять «как пришло» для отображения, но дедуп по casefold);
- пусто/None → `'Unknown'`.
- Чистая функция → покрывается unit-тестом.

### 3. Сервис синхронизации
`sync_user_clients(db) -> dict` (новый, напр. `app/services/client_sync_service.py`):
1. Постранично `get_all_hwid_devices()`.
2. Для каждого устройства: `app = parse_client_app(d['userAgent'])`; `panel_uuid = d['userUuid']`.
3. Резолв `panel_uuid → user_id`: по `User.remnawave_uuid`, фолбэк `Subscription.remnawave_uuid → user_id`. Предзагрузить карту uuid→user_id одним-двумя запросами (не N+1).
4. Собрать множество `(user_id, app, max last_seen)`; **upsert** в `user_clients`; строки, не встреченные в этом синке, — удалить (чтобы отписавшиеся клиенты уходили).
5. Вернуть счётчики `{devices, users, apps, duration}` и зафиксировать время последнего синка (в settings-кэше/таблице state — напр. простая запись в существующий механизм системных метрик или новое поле).
- Транзакционность/идемпотентность: синк безопасно повторять; ошибки панели — логировать, не падать целиком.

### 4. Планировщик + ручной запуск
- **Авто:** добавить вызов `sync_user_clients` в суточный цикл планировщика мониторинга (раз в 24ч; не на каждом тике). Гейт настройкой `CLIENT_SYNC_ENABLED` (default True) и интервалом `CLIENT_SYNC_INTERVAL_HOURS` (default 24).
- **Вручную:** `POST /cabinet/admin/broadcasts/clients/sync` (право `broadcasts:read`/admin) → запускает синк, возвращает счётчики + время. Защитить от параллельных запусков (lock/флаг).

### 5. Фильтр рассылки (бэкенд)
- `GET /cabinet/admin/broadcasts/filters` → добавить `client_filters: [{app_name, recipient_count, last_sync_at}]` — distinct `app_name` из `user_clients` + count пользователей (с учётом требований канала: для TG — есть `telegram_id` и не заблокирован, как в существующих фильтрах).
- Новая цель рассылки `client:<app_name>` (или отдельное поле `client_app` в запросе create/send/preview). Резолв получателей: `JOIN user_clients ON user_id WHERE app_name=<app>` + те же базовые условия канала, что и в других целях.
- `POST /preview` поддерживает подсчёт по этой цели.

### 6. UI (кабинет)
`AdminBroadcastCreate.tsx`:
- Группа выбора фильтра «По клиенту»: список приложений с количеством получателей (из `client_filters`).
- Кнопка «Обновить список клиентов» → `POST .../clients/sync`, спиннер, после — рефетч фильтров; показать «Обновлено: <время>».
- Выбор клиента передаётся как цель рассылки в create/send.

### 7. Границы / вне охвата
- Кто никогда не подключался (нет устройств) — ни в одном клиентском списке.
- Мульти-апп юзер — в каждом своём списке.
- Карточка пользователя с клиентом — НЕ в этом объёме.
- Email-канал: клиент — это про VPN-приложение; для email-рассылки фильтр по клиенту тоже применим (юзер тот же), но смысл основной — TG.
- Точность UA: «авто из данных», красивые имена/группировку синонимов (если один клиент шлёт разные UA) — не делаем; при необходимости позже.

## Тестирование
- Unit: `parse_client_app` — Happ/v2rayNG/Streisand/пусто/мусор.
- Синк (с моками `get_all_hwid_devices`): маппинг uuid→user, upsert, удаление устаревших, мульти-апп.
- Фильтр: `client_filters` считает корректно; резолв получателей по `client:<app>` даёт верный список (+канал).
- UI: type-check + build; список клиентов и кнопка синка.
