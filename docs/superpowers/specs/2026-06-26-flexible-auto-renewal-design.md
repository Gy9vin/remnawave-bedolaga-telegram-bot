# Гибкое авто-продление подписок — дизайн

- **Дата:** 2026-06-26
- **Статус:** согласован, готов к плану реализации
- **Репозитории:** бот (`remnawave-bedolaga-telegram-bot`) — основное; кабинет (`bedolaga-cabinet`) — UI карточки

## Проблема

1. **Баг:** авто-продление иногда не срабатывает.
2. **Видимость:** в карточке пользователя (админка кабинета) не видно — продлилось или нет.
3. **Цаденс:** при включённом автопродлении нужно за 3 дня до окончания пробовать продлить каждые 3 часа при положительном балансе.
4. **Гибкость:** продлевать «на максимум, что покрывает баланс», корректно учитывая число устройств и текущий режим (классика/тарифы). Примеры пользователя: пополнил 510 → должно продлить на 90 дней (а не на 30); пополнил 200 при 2 устройствах → 30д стоит 300 (200+100), не хватает — это надо видеть, а не молча падать.

## Текущее состояние (найдено разведкой)

Три перекрывающихся механизма автопродления:
- **A.** `_process_autopayments` — `monitoring_service.py:1415`, цикл каждые `MONITORING_INTERVAL` (60 мин), окно `autopay_days_before` (≤3д). Гейт `settings.ENABLE_AUTOPAY` — **дефолт `False`** (`config.py:443`). Выбор периода: `autopay_period_days` / `DEFAULT_AUTOPAY_PERIOD_DAYS` / shortest / 30.
- **B.** `_process_auto_renew_before_expiry` — `monitoring_service.py:1808`, за 15 мин до конца, **жёстко 30 дней** (`:1872`), флаг-guard `auto_renewed_before_expiry` (`models.py:2129`).
- **C.** `auto_purchase_saved_cart_after_topup` — после пополнения, требует Redis-корзину + `has_topup_intent`.

Ценообразование: единый `pricing_engine.calculate_renewal_price()` (`pricing_engine.py:521`) **авторутится** per-subscription: `tariff_id` есть → тарифный расчёт; иначе → классический (`_calculate_classic_core`, env-цены + `PRICE_PER_DEVICE`). Доп. устройства: `extra = device_limit - DEFAULT_DEVICE_LIMIT`, цена `extra * PRICE_PER_DEVICE * months` (классика `:830-836`; тариф `:648-657`).

Поля автопродления: `Subscription.autopay_enabled`, `autopay_days_before`, `autopay_period_days`, `auto_renewed_before_expiry` (`models.py:2123-2129`). **Нет** полей о последней попытке/результате — провалы нигде не персистятся, только логи.

Списание: `subtract_user_balance` (`crud/user.py:638`), `SELECT ... FOR UPDATE`, возвращает `False` если баланса нет.

## Решения (из обсуждения)

- При нехватке баланса — **продлевать максимум доступный период** (не настроенный, не гибрид).
- Сроки — **по готовым периодам режима** (классика: `AVAILABLE_RENEWAL_PERIODS`; тариф: периоды тарифа). **Без потолка** по `autopay_period_days` — всегда максимум по балансу.
- Объём — **точечные правки**, без слияния трёх механизмов в один.
- Списание за устройства — **аудит + фикс** расхождения «расчётная цена vs списанная сумма».

## Дизайн

### 1. Гибкий подбор периода — `select_affordable_renewal`
Новый хелпер (в `pricing_engine` или autopay-сервисе):
```
select_affordable_renewal(db, subscription, user) -> (period_days, price_kopeks) | None
```
- Источник периодов по режиму подписки: `subscription.tariff_id` есть → `tariff.get_available_periods()`; иначе → `settings.AVAILABLE_RENEWAL_PERIODS`.
- Перебор периодов **по убыванию**; для каждого цена через `calculate_renewal_price(db, subscription, period, user=user)` (единый источник, уже учитывает устройства и скидки в обоих режимах).
- Вернуть **первый период, цена которого ≤ `user.balance_kopeks`**. Если ни один не покрывается — `None` (попытка считается «недостаточно баланса», повтор через 3ч).

### 2. Корректное списание за устройства (фикс «не там перемножать»)
- Аудит пути автопродления/`extend`: сумма, **реально вычитаемая** из баланса, должна в точности равняться `calculate_renewal_price(...)` для выбранного периода (с устройствами). Никакого отдельного/повторного умножения на устройства или месяцы вне движка цен.
- Исправить найденные расхождения (если списание считается не через `calculate_renewal_price`).

### 3. Цаденс «каждые 3ч в окне 3 дня» (механизм A, точечно)
- Новое поле `Subscription.last_autopay_attempt_at` (nullable datetime).
- Попытка выполняется только если `last_autopay_attempt_at is None` или `now - last_autopay_attempt_at >= 3h` — независимо от `MONITORING_INTERVAL`. Поле обновляется при КАЖДОЙ попытке (успех/провал).
- Окно старта: подписка в пределах `autopay_days_before` (деф. 3) дней до `end_date`. Починить усечение целых дней (`(end_date - now).days`) → считать по часам/`total_seconds`.
- Повтор каждые 3ч пока: автопродление включено, подписка не продлена в этом цикле, баланс > 0, в окне.

### 4. Фиксы багов
- `ENABLE_AUTOPAY` — дефолт изменить на `True` (форк, автопродление требуется); задокументировать в `.env.example`. Опционально: startup-warning, если есть подписки с `autopay_enabled` но `ENABLE_AUTOPAY=False`.
- Механизм B: убрать хардкод 30 дней — использовать `select_affordable_renewal`; сбрасывать `auto_renewed_before_expiry` при каждом успешном продлении (подтвердить текущее поведение reset — агенты разошлись). Гарантировать идемпотентность между A и B (не списать дважды).
- Tariffs-режим без `tariff_id` и суточные — как и раньше вне охвата (явно залогировать пропуск).

### 5. Персист результата попытки
Новые поля на `Subscription`:
- `last_autopay_attempt_at: datetime | None`
- `last_autopay_status: str | None` — `success` / `insufficient_balance` / `error` / `skipped`
- `last_autopay_error: str | None`
- `last_autopay_renewed_at: datetime | None` (время последнего успешного автопродления)
- `last_autopay_period_days: int | None` (на сколько продлили)

Заполняются в механизме A (и B) на каждой попытке.

### 6. API (бот, кабинет)
- `UserSubscriptionInfo` (`app/cabinet/schemas/users.py:54`) — добавить: `autopay_days_before`, `last_autopay_attempt_at`, `last_autopay_status`, `last_autopay_renewed_at`, `last_autopay_period_days`.
- `get_user_detail` (`admin_users.py:674`) — отдать новые поля.

### 7. UI карточки (кабинет, `AdminUserDetail.tsx` блок подписки ~`:2233`)
Строка/блок «Автопродление»:
- Бейдж вкл/выкл (`autopay_enabled`) + «за N дней» (`autopay_days_before`).
- Последняя попытка: время + результат — ✅ продлено (с датой и периодом) / ⚠️ недостаточно баланса / ❌ ошибка / — нет попыток.
- Стиль — как соседние строки деталей подписки.

### 8. Миграция
Alembic-миграция: новые колонки на `subscriptions` (поля из §3 и §5). Все nullable, без бэкфилла.

## Граничные случаи
- Баланс не покрывает даже минимальный период → `insufficient_balance`, повтор через 3ч, уведомление по существующей `AutopayFailState`-логике (не спамить).
- Нулевая цена периода → пропуск с логом (как сейчас), не «бесплатное» продление.
- Модем-аддон (`device_limit` +1 от модема) — не считать как платное устройство дважды (учтено в `_effective_device_limit_with_modem`, `pricing_engine.py:576`).
- Подписка с `tariff_id`, но не загруженным `tariff` → fallback в классику (как сейчас).

## Вне охвата
- Слияние трёх механизмов в один движок.
- Email-уведомления о попытках.
- Суточные тарифы (свой сервис).
- Прошлые подписки без новых полей — просто покажут «нет данных» до первой попытки.

## Тестирование
- Unit: `select_affordable_renewal` — классика (30/90/180/360 из конфига) и тариф; с доп. устройствами; баланс ровно на границе; баланс ни на что не хватает → None.
- Соответствие списанной суммы и `calculate_renewal_price` (устройства).
- Цаденс: попытка не чаще 3ч; окно 3 дня по часам.
- Идемпотентность A↔B.
- API/UI: новые поля доходят и рендерятся.
